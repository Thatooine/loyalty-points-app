Section 01

What I built

A loyalty-points ledger exposed as a JSON-RPC 2.0 API, backed by PostgreSQL.

👤Accounts

Opened for a user at registration; readable, renameable, and — for admins — adjustable.

💰Earn / Spend

Recorded against an account, with the current balance tracked at all times.

🔁Idempotency

The same transaction ref never counts twice — the constraint is the dedupe.

🛡️Overdraft floor

A spend can never drive a balance below zero — enforced atomically.

🔑Access control

member and admin roles in two layers; crediting is operator-only.

⏱️Sessions

Short-lived (1h) JWTs with server-side revocation — logout invalidates every token.

📦Batch ingestion

A CLI ingests a CSV of transactions safely and reports a summary.

📋Audit trail

Every attempt — accepted, duplicate, or rejected — is recorded.

✅Fully tested

Unit, adaptor, validation, and black-box HTTP integration coverage.

Section 02

Key technology choices

Every choice is framed the same way: what it buys and what it costs. Nothing here is free.

PostgreSQL, not SQLite +

The assignment recommends SQLite and allows Postgres. The correctness story for concurrent writes is the interesting part of this problem.

Gain

Real row-level locking and a guarded-UPDATE concurrency model without SQLite's single-writer caveats. Two overlapping spends serialise on the row, not on a process-wide lock. Plus CHECK constraints, composite indexes for keyset pagination, and GENERATED ALWAYS AS IDENTITY.

Trade-off

A real network dependency and a container to run. Bought back with docker compose up + baked-in defaults — but a reviewer wanting plain go run pays a small tax SQLite wouldn't charge.

JSON-RPC 2.0 over gorilla/rpc/v2 +

A loyalty ledger is a set of verbs, not resources: EarnPoints, SpendPoints, Logout. These are commands that map cleanly onto named methods — REST would mean inventing resource nouns and arguing over whether "spend" is a POST or a PATCH.

Gain

Declarative wire layer (methods auto-register by signature); one choke point for auth + logging; the policy table is a literal, auditable attack surface that fails closed; transport-agnostic uniform error shape; batches are first-class.

Trade-off

Gives up HTTP-native semantics — everything is POST, errors come back as HTTP 200. But this ledger has almost no cacheable reads, so HTTP caching buys little. The one papercut: the codec won't decode JSON-RPC batch arrays — which pushed batch ingestion toward the single ordered payload I preferred anyway.

Ports and adapters +

pkg/<domain> holds interfaces, DTOs, Validate(), and adaptors; internal/pkg/<domain> holds the Postgres implementations. Production code depends only on the pkg interfaces.

Gain

Testable with mocks at the port boundary (most of the suite needs no DB), SQL isolated to one layer, swapping an implementation is a one-line wiring change. The internal/ boundary stops accidental concrete imports.

Trade-off

More files and indirection (interface + DTO + impl + mock) per capability. Real ceremony at this size — leaned on .claude/skills/ scaffolds to keep boilerplate consistent.

pgx/v5 via the database/sql stdlib interface +

The driver is registered through pgx's stdlib shim rather than its native API.

Gain

Keeps the standard *sql.DB/*sql.Tx abstractions — what the context-resolved executor and TxManager are built on — while getting pgx's well-maintained driver and pool. Repositories aren't coupled to pgx types.

Trade-off

Forgoes pgx's native niceties (richer type mapping, COPY, batch pipelining). None needed here; stdlib portability was worth more.

JWS / RS256 via go-jose, not symmetric HS256 +

Access tokens are asymmetrically signed.

Gain

Only the private-key holder can mint tokens; any service verifies with the public key. The right shape for later splitting issuer from verifier — a leaked verification key can't forge tokens. go-jose is mature and audited.

Trade-off

Heavier than HMAC and brings asymmetric-key management. For a single-process deployment HS256 would be simpler — paid for the verification model deliberately, anticipating split trust domains.

bcrypt for password storage +
Gain

Adaptive, salted hashing with the salt embedded in the digest — no separate salt column, and the work factor can be raised over time. The conservative, well-understood default.

Trade-off

Deliberately slow and caps input at 72 bytes; a memory-hard KDF (argon2/scrypt) resists GPUs better. bcrypt was the lower-risk, battle-tested pick for a take-home.

zerolog for structured logging +
Gain

Zero-allocation JSON logs plus a context-bound logger: the middleware mints a request_id and binds it so every downstream log.Ctx(ctx) line is correlated for free.

Trade-off

A specific logging dependency threaded through the code. slog would cover much of this now — the choice I hold most loosely — but context-bound correlation was the deciding feature.

cobra + viper for the CLI and config +
Gain

cobra gives a real command tree, flags, and help with little code; viper gives layered config (env-var over baked-in defaults) so the server runs out of the box yet every secret is overridable.

Trade-off

Heavy for one subcommand and two config keys. Took the headroom because ingestion grows subcommands, and viper's env binding is the 12-factor seam a real deployment wants.

Hand-rolled embedded migrations, no library +

Numbered *.sql files are go:embed-ed and applied in lexical order on startup and in tests.

Gain

Schema travels inside the binary, the same code path runs in production and the per-test fresh-schema harness, and evolving the schema is "add the next numbered file." Zero operational moving parts.

Trade-off

No down-migrations, checksums, or out-of-order detection. Fine for a forward-only take-home; a long-lived service would want golang-migrate/goose.

Section 03

The invariants, and where they live

Every ledger write flows through one method — WalletServiceImpl.ProcessTransaction — so the invariants are enforced and tested in exactly one place. EarnPoints, SpendPoints, and ProcessTransactionBatch are thin wrappers.

Idempotency

The constraint is the dedupe

The ledger insert is attempted first; transactions.ref carries a UNIQUE constraint. A duplicate fails on the constraint and the original outcome returns with duplicate = true. Deliberately not check-then-insert — that has a race window; the unique index does not.

Overdraft floor

Read-check-write in one statement

Balance is mutated with a single guarded statement, so two overlapping spends cannot both pass the check against the same starting balance. A table-level CHECK (balance >= 0) backstops it.

UPDATE accounts SET balance = balance + $delta
WHERE id = $id AND balance + $delta >= 0

Zero rows affected means either insufficient balance or a missing/unowned account.

Rejected

The obvious SELECT-balance → compare-in-Go → UPDATE opens a race window: two concurrent spends read the same balance, both pass, both write — negative. Pushing the check into the WHERE collapses read-check-write into one atomic statement and lets Postgres serialise contenders on the row, no explicit lock or higher isolation level needed.

Concurrency safety

A unit of work via context-resolved executor

Repositories never hold a *sql.Tx. Each resolves its executor from the context (pkgSQL.ExecutorFromContext): if TxManager.RunInTx placed a transaction on the context, the repository uses it; otherwise the pool. The caller decides what is atomic, and the same repository code works standalone or composed.

Rejected

Threading a *sql.Tx through every method signature — that forces each method into two flavours (pool vs tx) or leaks the transaction into the signature even when it runs standalone. Stashing the executor on the context lets each repository write one query path against a small Executor interface, with RunInTx deciding atomicity from outside. The cost: the transaction travels implicitly on the context — slightly more magic — but the repository layer stays free of transaction bookkeeping.

Audit trail

Survives rollback when it must

Accepted and duplicate attempts are audited inside the unit of work (committing with the ledger row). Rejected attempts are audited on the plain context (the pool), so the audit row survives the rolled-back transaction — a rejection is still recorded even though nothing else was written.

Rejected

Writing every audit row inside the same transaction was wrong precisely for rejections: rolling back the overdraft would roll back its own audit record, silently losing every failed attempt. Splitting the rejected write onto the pool is what keeps "someone tried to overspend" durable.

Section 04

Access control

Two roles, enforced in two independent layers.

Token shape, storage, and validation

Access is carried by a signed JWT (JWS, RS256, compact serialization via go-jose). The payload is the LoginClaim:

{
  "userID": "0c5f...",
  "email": "rina@example.com",
  "role": "member",
  "permissions": ["account:read:own", "wallet:transact:own"],
  "expirationTime": 1718534400,
  "tokenVersion": 0,
  "lastName": ""
}
  • Issued at register and login, 1h expiry. Permissions resolved from role at issue time and embedded — no per-request DB lookup for permissions.
  • Stored client-side; no server-side session table. Presented as Authorization: Bearer or an access_token cookie.
  • Validated by verifying the RS256 signature, checking expirationTime, and comparing the claim's tokenVersion against the user's current value. Signature verification means embedded permissions can't be tampered with.

Session revocation (logout)

A signed JWT is self-contained, so a leaked token would stay valid until expiry. Each user row carries a token_version (session epoch), stamped into every token and re-checked on every protected request. Session.Logout increments it — invalidating every token in one step ("log out everywhere"). Login deliberately does not bump it, so concurrent device sessions coexist. Cost: one indexed read per protected request.

1

Method gate — all-or-nothing

authorizationMiddleware reads the JSON-RPC method out of the request body and consults a Policy mapping method → permissions, plus a public-method set. Public methods (register, login) pass through. New protected methods must be added to DefaultPolicy() or they're rejected — fail-closed by default.

This is deliberate over the easy alternative of guarding each handler individually (if !authorized { … }): a per-handler check is something you can forget when writing a new method, and the failure mode of forgetting is an open endpoint. Here the failure mode is a closed one — the method returns "method not allowed" until it's listed — so the unsafe mistake is structurally impossible and the safe mistake is loud.

2

Ownership scope — own vs all

Breadth is enforced in the data layer, on demand: a repository calls authorization.IsGranted(ctx, Perm…All), and without the :all permission the SQL is scoped to owner_id = $UserID. A non-owner gets ErrNotFound — indistinguishable from genuinely missing, so ownership is enforced without leaking existence.

Permissions are resource:action:scope strings; roles map to a fixed, no-wildcard set. Crediting points needs the all-scoped wallet:transact:all that only admins hold; a member's wallet:transact:own unlocks SpendPoints against their own account only — a member cannot mint points. The acting UserID is always taken from the verified claim, never the client payload.

How a user becomes an admin

Registration always creates a member — no self-service path to admin. An admin is provisioned out-of-band:

UPDATE users SET role = 'admin' WHERE email = 'ops@example.com';

For local dev, go run ./cmd/bootstrap resets the DB and recreates a well-known admin (system@mail.com) — a dev convenience and clean-slate reset, not a production path.

Section 05

Batch ingestion and ordering

The CLI (loyalty-cli ingest) is a thin client; all business rules live on the server.

It parses the CSV, sorts rows by occurred_at then line (a stable sort with the 1-based data line as tiebreaker; a blank occurred_at sorts first), and sends the whole batch as one ordered JSON-RPC request to the admin-only Wallet.ProcessTransactionBatch.

Why one ordered request

The JSON-RPC spec lets a server process a batch in any order and concurrently, and the gorilla json2 codec doesn't decode batch arrays at all. Because the overdraft floor makes each write order-dependent (an earn must precede the spend it funds), ordering must be guaranteed — so the batch travels as a single ordered payload applied sequentially.

Ordering enforced server-side

Sorting in the CLI alone would leave the invariant at the mercy of the client. So ProcessTransactionBatch re-sorts by occurred_at itself, making correct chronology a server property. Two details were deliberate: it sorts a copy so the caller's slice isn't mutated and results can still map to the sent order; and the sort is stable (submission order as tiebreaker) so equal-timestamped rows can't reorder run-to-run and strand an earn after the spend it funds. The CLI's sort is now belt-and-braces — it still powers the --dry-run preview.

Because the server may reorder, per-element results come back in applied order and callers correlate them by ref, not position. Reprocessing the same file is safe by construction: idempotency dedupes on ref, so re-ingestion yields duplicates, not double counts. The server returns per-element outcomes (accepted / duplicate / rejected with a reason) plus summary tallies; the CLI prints processed / accepted / duplicates / rejected.

Section 06

Data model

⏱️ Timestamps

Stored as RFC3339Nano UTC TEXT — human-readable and lexicographically sortable, so ordering and keyset pagination rely on plain string comparison.

🔗 OwnerID

A foreign key to users.id, denormalized onto transactions and audit_log so entries are attributable without a join. The DTO field is UserID; the SQL column is owner_id.

📜 Schema

Lives in pkg/postgres/migrations/*.sql, embedded and applied in lexical filename order on startup and in tests.

Section 07

Trade-offs and what I'd do next

The single inventory of what I deliberately did not build — the out-of-scope notes scattered above all resolve here. Each was a conscious cut to keep the focus on the correctness-critical core, not an oversight.

ADMIN

Admin provisioning is manual (SQL promotion). Fine for a take-home; a real system needs an admin-bootstrap path or invite flow. I chose manual promotion over an admin-creates-admin RPC specifically to avoid shipping a privilege-escalation surface I couldn't fully harden in scope.

WALLETS

One wallet per user. Registration opens a single default account. The data model already supports multiple (accounts.owner_id), so a CreateAccount RPC is a small addition — left out to stay in scope.

TOKENS

Permissions are snapshotted into the token. A role change takes effect on next login (≤1h). Tokens are revocable via token_version (logout-everywhere). I chose not to build per-device revocation because it needs per-token tracking (a server-side token table), trading away the stateless-JWT property the auth model is built on; the epoch counter buys "log out everywhere" for one indexed read instead.

SCALE

No rate limiting / request size limits on ingestion. The batch applies in a single unit of work — simple and correct, but not ideal for very large files; chunking would be next.

MIGRATIONS

Forward-only migrations. No down-migrations, checksums, or dirty-state tracking. I decided against golang-migrate/goose because embed-and-apply-on-startup has zero operational moving parts and suffices for a forward-only take-home; a long-lived service would outgrow it.

Section 08

AI workflow

This section documents how AI tooling was used, per the assignment.

I used Claude Code as a pair-programmer throughout, with a deliberately plan-first, test-backed workflow rather than free-form generation.

Conventions encoded as reusable skills

Rather than re-explain the architecture in every prompt, I captured the recurring, error-prone tasks as Claude Code skills in .claude/skills/ — each a checklist plus canonical reference, so generated code mirrors the existing patterns instead of drifting.

SkillWhat it doesLeaves out
code-structureThe arbiter: decides what belongs in adaptor vs service vs repository, and routes any new write path through the single owner of that mechanic.—
scaffold-crud-repositoryFrom one entity struct, generates the whole persistence layer — interface + DTOs, Validate(), the Postgres impl, the mock, and validation tests.adaptor, policy, wiring
scaffold-serviceFrom a capability interface, generates the impl orchestrating repositories in one unit of work, its adaptor, mock, Validate(), and unit tests. Writes no SQL.repository SQL, policy, wiring
add-rpc-methodMakes a method callable end-to-end: interface, DTO, adaptor, SQL impl, and the DefaultPolicy() entry — the one omission that silently breaks a method.—
go-unit-testsPure-Go tests: mock-first convention, table-driven validations, and the non-negotiable happy-path-and-each-error-branch rule.—
endpoint-integration-testBlack-box HTTP/JSON-RPC tests asserting both wire response and persisted rows, plus mandatory negative cases.—

How they compose

code-structure decides the shape; the two scaffold-* skills generate a layer each (stopping short of the wire so the division stays honest); add-rpc-method exposes it and closes the policy gap; the two test skills cover it at unit and endpoint level. Each skill's "out of scope" list points at the next — so the model hands work off rather than half-doing a neighbouring layer.

How I steered the rest

I accepted the model's mechanical scaffolding, SQL boilerplate, and table-driven test stubs readily. I pushed back on / edited the correctness-critical calls: the concurrency model (single guarded UPDATE over check-then-insert), the batch transport (one ordered request, not a JSON-RPC batch array, because ordering must be guaranteed), and audit-on-rollback (rejected attempts must survive). I owned those decisions and used the model to implement and test them rather than to decide them — so AI accelerated the breadth while the load-bearing decisions stayed human-reviewed.
